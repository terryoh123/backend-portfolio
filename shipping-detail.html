택배사 송장 자동발번·출력·이력관리 시스템

택배 API 연동 / 비동기 처리 / 대량 전송 안정화 / 이력관리

여러 택배사 API와 연동하여 송장 자동발번, 출력, 전송 결과 이력관리를 처리하는 시스템을 개발하고 유지보수했습니다. 단순 API 호출이 아니라, 대량 주문 처리 상황에서도 안정적으로 동작할 수 있도록 비동기 처리, 재시도 로직, 실패 건 재처리, 결과 로그 저장 구조까지 개선한 프로젝트입니다.

PHP CodeIgniter REST API Shell Script Retry Logic Logging

Project Highlights

30분 → 3분

처리시간 단축

1000건 이상 동시 처리 상황에서 병목을 줄이고 처리 속도를 크게 개선

70% → 95%+

성공률 개선

비동기 처리, 재시도 로직, 분산 대응으로 대량 발번 성공률을 향상

실패 이력 관리

운영 대응 강화

실패 건 조회, 원인 확인, 재전송이 가능한 운영 기능으로 고객사 대응력 개선

Problem

배경
CJ대한통운, 우체국, 한진, 로젠, 롯데, BGF 편의점택배, 카카오T 당일배송 등 여러 택배사와 연동해 송장 자동발번/출력 기능을 제공하는 시스템을 운영했습니다.
문제 상황
  • 명절 시즌과 같은 주문 집중 시기에 대량 발번 요청이 한 번에 몰리며 무한 로딩과 타임아웃 발생
  • 순차 처리 구조로 인해 대량 건 처리 속도가 느리고 서버 병목이 발생
  • 택배사 API 호출 제한(Rate Limit)을 넘기면서 실패율 증가
  • 실패 건 확인과 재전송이 어려워 운영팀과 고객사 대응 부담이 컸음
원인 분석
  • 단일 스레드 순차 처리 구조
  • 택배사별 API 호출 제한에 대한 제어 부족
  • 서버 리소스 부족 및 병목 심화
  • 실패 결과를 운영자가 직접 추적하기 어려운 구조

Solution

해결 방식
  • 쉘 스크립트를 활용해 비동기 처리 구조를 도입하고 순차 처리 병목을 완화
  • 택배사 API 초당 호출 제한을 고려한 Retry Logic 및 재시도 제어 적용
  • 서버 부하를 분산시키는 방식으로 대량 요청 대응 구조 개선
  • 실패 건 자동 재처리 배치를 구성해 사후 복구 가능하도록 설계
  • 결과 로그 저장과 이력관리 화면을 추가해 실패 사유 확인 및 재전송이 가능하도록 개선
운영 관점에서 개선한 부분
  • API 인증키, 송장 옵션 등 택배사별 설정을 관리자 화면에서 관리 가능하게 구현
  • 전송 결과와 실패 사유를 이력 화면에서 바로 조회할 수 있도록 구성
  • 사용자가 실패 건을 직접 확인하고 재전송할 수 있도록 운영 기능 제공
성과
  • 1000건 이상 동시 처리 시 처리시간을 약 30분에서 3분 수준으로 단축
  • 발번 성공률을 70% 수준에서 95% 이상으로 개선
  • 고객사 클레임을 줄이고, 대량 전송 상황에서도 신뢰할 수 있는 시스템으로 고도화

Implementation Summary

1. 설정 관리
택배사 계약코드, API 인증키, 송장 옵션 등 택배사별 연동에 필요한 설정값을 관리자 화면에서 관리하도록 구현
2. 송장 발번 요청 처리
대량 주문 데이터를 기준으로 택배사 API에 발번 요청을 보내고, 요청 결과를 수집하는 로직 구성
3. 비동기 처리 및 재시도
순차 처리 병목을 줄이기 위해 비동기 처리 구조와 재시도 로직을 도입하여 실패율과 지연을 개선
4. 결과 이력 저장
성공/실패 여부와 원인, 재전송 상태 등을 이력관리 페이지에 남겨 운영자가 직접 추적 가능하도록 구성

송장 옵션 세팅 화면: 발번 방식, 운송장 출력 옵션 등을 운영 상황에 맞게 설정 가능

비동기 처리 코드: 대량 발번 시 병목을 줄이기 위한 비동기 전송 및 요청 분산 구조
API 실행 및 로그 저장 코드: 택배사 연동 결과를 저장하고 이력관리 화면과 연결
이력관리 컨트롤러: 결과 조회, 검색, 재전송 연계 등 운영 기능을 담당

What I Learned

  • 대량 처리 환경에서는 단순 기능 구현보다 처리 구조와 병목 제거가 훨씬 중요하다는 점
  • 외부 API 연동은 성공/실패 자체보다 재시도, 추적, 운영 대응 구조까지 함께 설계해야 한다는 점
  • 운영 시스템은 “잘 동작하는 것”뿐 아니라 “문제 발생 시 빠르게 원인을 찾을 수 있는 것”이 중요하다는 점
